# Known Issues

<Callout type="warning">
pgGraph is currently alpha software. Do not use it for production workloads
until the next correctness, SQL contract, persistence, and operational
hardening update lands.
</Callout>

<Callout type="info">
pgGraph is built around a simple launch standard: the most critical correctness,
durability, safety, and operational paths should be solid first. There is always
another way to optimize a graph engine, especially inside PostgreSQL, so we are
launching with the critical paths working and then continuously improving in
the open.
</Callout>

This page documents known alpha limitations, next-update hardening work, and
lower-priority follow-up items. The list is ranked by expected user impact, not
by implementation difficulty.

The tracked execution plan for higher-priority open rows is
`known-issues-completion-plan.md`. This page remains the authoritative issue
register; the plan records implementation order, test gates, and regression
risks for the same rows.

Low-level PostgreSQL extension work involves real tradeoffs. Some optimizations
make a query faster by keeping more data in memory, while others reduce memory
pressure by doing more work per request. pgGraph has to weigh those choices
carefully because it runs inside PostgreSQL backend processes rather than a
separate service with its own resource envelope.

## Next Update Scope

The next update is focused on issues that are either hard to change after users
depend on SQL behavior, or important enough to fix before recommending
production use.

### Completed In This Hardening Pass

These items were completed and removed from the active limitation tables. The
remaining sections below are still open work.

| Area | Completion |
|---|---|
| Build advisory lock keys | Build and vacuum advisory lock keys now use named constants and a shared SQL helper. |
| Unsigned filter checks | Unsigned in-memory filter checks are separated from signed typed-filter conversion, with benchmark evidence recorded privately. |
| Job status values | Build and maintenance job status strings are centralized through a typed `JobStatus` enum. |
| Discovery edge registration | Auto-discovery and targeted discovery share edge-label and registered-edge construction helpers. |
| Discovery OID modeling | Discovery internals use explicit optional OIDs instead of a missing-OID sentinel. |
| Relation lookup errors | Missing relation lookups now report a controlled graph error instead of leaking raw `regclass` behavior. |
| Test locality | Pure Rust helper tests were moved beside their owning modules and `unit_tests.rs` was removed. |
| Edge-store builders | Sorted CSR construction now routes through `SortedEdgeStoreBuilder` instead of duplicating the state machine. |
| Persistence byte decoding | `.pggraph` scalar and length-prefixed payload decoding now use shared helpers. |
| Internal sync SQL | Sync status, tenant setting, and table-OID SPI reads bind data values through parameters. |
| Persistence section writing | `.pggraph` section offsets, scalar arrays, and length-prefixed payloads now use writer helpers. |
| Date and timestamp filters | Date and timestamptz structured-filter casts now bind input text through SPI parameters. |
| Composite-key and property-column modeling | Registered primary-key and property-column data now crosses Rust build, discovery, sync, and trigger-generation paths through typed column-set structs instead of ad hoc comma splitting. |
| Background worker metadata | Build and maintenance background workers now pass typed JSON metadata instead of fragile delimiter-separated strings. |
| Filter catalog lookup SQL | Structured-filter catalog lookup now binds filter column values through SPI parameters. |
| Node reference input | Traversal node-reference parsing now accepts normal JSON arrays and objects while preserving `graph.node_ref_string()` text compatibility. |
| Native identifier quoting | Dynamic identifier fragments now use pgrx's PostgreSQL-backed identifier quoting outside pure Rust tests. |
| PostgreSQL API boundary wrappers | Background-worker notify PID access now goes through a small wrapper, and remaining direct `pg_sys` usage is limited to extension ABI, ACL, OID, mmap, and PostgreSQL data-directory boundaries. |
| Traversal uniqueness | Multi-start `graph.traverse()` and `graph.traverse_search()` now honor `node_global` by de-duplicating reached nodes across roots before final pagination, while `node_per_root` keeps per-root results. |
| Path aggregation | `all_possible_paths` counting and aggregation now keep enumerated paths as shared indexed snapshots, avoiding duplicate full-path clones and coordinate materialization for path-count checks. |
| Filter model and legacy numeric filters | SQL traversal now builds pushdown predicates only from structured typed filters; the legacy unsigned condition parser is limited to tests, fuzzing, and development helpers so signed values cannot be reintroduced through the public traversal path. |
| Dynamic table SQL | Build and estimate reltuples reads now resolve registered table names through bound OID lookups, while table and column SQL fragments remain limited to validated catalog/regclass names and quoted identifiers. |
| Discovery SQL | Auto-discovery schema values, targeted table OIDs, primary-key exclusions, and targeted foreign-key table sets now use SPI parameters instead of generated literal or `IN (...)` SQL fragments. |
| Truncate handling | Sync replay now maintains table-to-node membership bitmaps, so table truncation tombstones only nodes from the affected table and removes their tenant membership. |
| Tenant bitmap mutation | Sync replay now uses old and new tenant values from row images to mutate only the affected tenant bitmaps for update, delete, and primary-key replacement. |
| Internal scheduler | Scheduler ownership is documented as external: pgGraph exposes the admin-only `graph.run_scheduled_maintenance()` target, while pg_cron, Kubernetes CronJobs, systemd timers, Docker init SQL, or app schedulers own cadence and lifecycle. |
| Background job failures | Build and maintenance workers now record failed status and error detail from a fresh worker transaction after the work transaction returns, and failure recording does not overwrite completed jobs. |
| Artifact writes | `.pggraph` writes now stream fixed sections, primary-key offsets, and primary-key bytes through a seekable artifact writer that backpatches the header and CRC while preserving temp-write, fsync, atomic rename, and reload-validation behavior. |
| Persistence I/O | Background build and maintenance jobs now expose persistence write/fsync and reload-validation phases in status rows, while artifact writes retain synchronous durability and check PostgreSQL interrupts during long section loops. |
| DFS neighbor expansion | DFS traversal now expands base and overlay neighbors in reverse stack order without materializing a full per-node neighbor vector, while preserving overlay de-duplication and traversal result order. |

## P1: Query Performance Follow-Ups

These items are expected to improve latency or memory pressure, but they are
not changes to the public SQL contract.

| Area | Current limitation | Planned direction |
|---|---|---|
| Traversal internals | Some traversal paths still have avoidable per-query or per-node allocation. | Benchmark sparse metadata and reusable scratch buffers, keeping the dense path where it remains faster. Completion plan: `known-issues-completion-plan.md`. |
| Edge overlays | Deleted overlay checks and overlay construction can add hot-path hash lookups. | Add cache-friendly or oriented overlay representations only after overlay-size benchmarks define the threshold where they win. Completion plan: `known-issues-completion-plan.md`. |
| Connected components | Component helpers recompute sizes and can materialize full row vectors before slicing. | Reuse component-size data and delay row materialization until after filtering, sorting, and pagination where possible. Completion plan: `known-issues-completion-plan.md`. |
| Reverse graph build | Reversing a CSR graph can allocate intermediate edge structs and sort them. | Add a linear two-pass reverse CSR builder only if build-memory benchmarks justify another code path. Completion plan: `known-issues-completion-plan.md`. |
| Build-time SPI setup | Some build and discovery paths open repeated SPI contexts or run per-table metadata queries. | Batch metadata reads and reuse SPI contexts where query shape remains straightforward and errors stay precise. Completion plan: `known-issues-completion-plan.md`. |
| Source search recheck | Search still verifies source-row matches in Rust after SQL candidate filtering and can allocate per candidate row. | Reduce per-candidate allocation while preserving SQL/Rust predicate parity through shared fixtures. Completion plan: `known-issues-completion-plan.md`. |
| Hydration setup | Batched hydration can resolve all registered table OIDs before knowing which tables are needed. | Resolve only table OIDs needed by the requested page and reduce SPI context churn. Completion plan: `known-issues-completion-plan.md`. |
| Aggregation hydration | Some aggregation paths clone node IDs, JSON values, roots, and table names for lookup ownership. | Use borrowed lookup shapes internally where pgrx row lifetimes allow it, with parity tests for result JSON. Completion plan: `known-issues-completion-plan.md`. |
| Resolution delta lookups | Post-build sync inserts can accumulate in a delta structure that is scanned before the finalized index. | Add an indexed delta map only if sync-delta growth benchmarks show material lookup cost. Completion plan: `known-issues-completion-plan.md`. |

## P2: SQL Ergonomics and Documented Semantics

These items clarify behavior or broaden the API. They are planned, but should
not block use of the current documented SQL surface.

| Area | Current limitation | Planned direction |
|---|---|---|
| Auto-discovery text properties | Auto-discovery currently restricts discovered property columns to string-like values of bounded length. | Document the rule clearly and consider making the policy configurable. |
| Structured filter operators | Structured filters currently focus on equality, inequality, range comparisons, and `between`. | Add `in`, `not_in`, text search, and explicit null operators as product requirements demand. |
| Legacy raw filters | The legacy raw filter parser is compiled only for tests, fuzzing, and development helpers. | Delete the development-only parser after fuzz coverage moves fully to structured filters. |
| Case-insensitive search | `lower(src.col::text)` needs matching expression indexes for good performance. | Document recommended indexes and consider helper DDL. |
| GUC enum behavior | Some typed Rust settings are exposed as string GUCs to preserve aliases and warning behavior. | Consider native PostgreSQL enum GUCs only with an explicit compatibility plan. |

## P3: Internals and Maintainability

These are mostly contributor-facing improvements. They help long-term velocity,
but they do not change core user behavior.

| Area | Current limitation | Planned direction |
|---|---|---|
| SQL facade and pg test layout | Some SQL facade and pg test files are included textually from `lib.rs`. | Replace `include!` usage with normal modules and local imports. |
| Build/vacuum pipeline | Build and vacuum share a lot of pipeline logic. | Extract a shared helper while keeping build-only and vacuum-only behavior explicit. |
| Engine state encapsulation | Some internal fields with invariants are still mutated directly. | Encapsulate state such as `built`, `sync_status`, `is_read_only`, `needs_vacuum`, mmap, and resolution state behind validated helpers. |
| Mmap mutation API | Direct mutation on mmap-backed stores is not consistently represented. | Route mutations through engine helpers or make direct store mutation fail consistently. |
| Mmap resolution state | Mmap resolution offsets and backing storage are split across engine fields. | Couple the backing data with the `MmapBacked` variant so invalid states are harder to represent. |
| Filter op shape | Filter variants carry repeated column target data. | Refactor toward a shared column target plus a condition enum. |
| Sync row parsing | Some sync paths parse the same row JSON more than once. | Parse once at the boundary and pass borrowed row values through replay helpers. |
| Sync mutation coupling | Some edge overlay maintenance relies on caller ordering around node replacement. | Encapsulate node and edge replay behind a typed sync-row operation helper. |

## Lower-Priority Optimizations

These are intentionally deferred until measurement shows they matter:

- Avoiding small string allocations in config parsing and UUID parsing.
- Replacing bitmap serialized-size estimates with native roaring helpers if the
  pinned crate exposes them.
- Reducing low-level clone counts in builder, traversal, weighted path, and
  aggregation paths.
- Combining duplicate path reconstruction passes after the larger traversal
  allocation work is complete.
- Cleaning up long internal function signatures when the surrounding traversal
  parameters stabilize.
- Replacing manual pointer alignment checks with newer standard-library helpers
  once the supported Rust version makes that clean.
- Cleaning up padding-bit handling in `active_count()`.
- Moving from custom parse helpers to `FromStr` for SQL-facing enums.
- Introducing production newtypes for node indices and edge type IDs after
  benchmarking the hot paths.
